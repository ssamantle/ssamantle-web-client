# 에이전트 작업 가이드라인

본 문서는 이 프로젝트에서 작업을 수행하는 모든 AI 에이전트가 준수해야 할 작업 절차와 규약을 정의합니다.

## 1. 작업 환경 및 워크트리 구성

에이전트는 메인 작업 디렉터리의 안정성을 위해 독립적인 Git 워크트리(Worktree) 환경에서 작업합니다.

* **작업 위치:** 모든 작업은 반드시 현재 저장소 루트를 기준으로 `./worktrees` 디렉터리 아래에서 수행합니다.
* **워크트리 명명 규칙:** `feature/<작업-요약>` 형식을 사용합니다.
    * *예: `worktrees/feature/user-auth-hook`*
* **절차:** 작업을 시작하기 전 해당 디렉터리를 생성하고 이동한 뒤 개발을 진행합니다.

## 2. 설계 및 계획 프로세스 (구현 전 단계)

사용자로부터 기능 구현 요청을 받으면 코드를 작성하기 전, 반드시 아래의 **[분석 -> 설계 -> 검토]** 단계를 거칩니다.

### 단계 1: 분석 및 기술 검토
* **컴포넌트/훅:** 필요한 React 컴포넌트 구조와 커스텀 훅의 역할을 정의합니다.
* **비즈니스 로직:** 상태 관리 및 데이터 흐름을 검토합니다.

### 단계 2: 구현 및 커밋 메시지 설계
* **단계별 계획:** 구현 과정을 논리적인 단위로 쪼개어 단계별 계획을 세웁니다.
* **커밋 메시지 계획:** 각 단계에 대응하는 커밋 메시지를 설계합니다.
    * **규약:** Conventional Commits (`feat:`, `fix:`, `refactor:` 등)
    * **언어:** 반드시 **한국어**로 작성합니다.

### 단계 3: 설계 최적화 (Self-Review)
* 초안에서 불필요하거나 과하게 설계된(Over-engineered) 부분이 없는지 확인합니다.
* 기존 프로젝트 컨벤션과 일치하는지 검토 후 계획을 수정/보완합니다.

## 3. 사용자 보고 및 승인

설계가 완료되면 사용자에게 다음과 같은 형식으로 보고하고 승인을 받습니다.

> **📋 구현 계획 제안**
> 1. **분석 결과:** (필요한 요소 기술)
> 2. **단계별 구현 계획:**
>    - [ ] 1단계: ... (커밋 계획: `feat: ...`)
>    - [ ] 2단계: ... (커밋 계획: `feat: ...`)
> 3. **설계 검토 의견:** (최적화 및 수정된 사항 기술)

## 4. 작업 수행 및 결과 보고 (구현 단계)

사용자의 승인 후 각 단계를 수행할 때마다 다음 원칙을 지킵니다.

* **진행 보고:** 현재 어떤 단계를 완료했는지 명확히 알립니다.
* **자동 커밋 수행:** **각 단계의 작업을 마칠 때마다**, 사용자에게 별도 요청받지 않아도 해당 단계 변경사항을 즉시 커밋합니다.
    * **예외:** 실제 파일 변경이 없는 단계는 커밋하지 않고 그 사실을 보고합니다.
    * **범위:** 해당 단계에서 발생한 변경 파일만 스테이징하여 커밋합니다.
* **커밋 메시지 공유:** 자동 커밋 후, 사용자가 이력을 바로 확인할 수 있도록 커밋 해시와 메시지를 함께 보고합니다.
    * **형식:** `feat: <내용>`, `fix: <내용>` 등 (한국어 사용)

---
> **출력 예시:**
> "X단계 작업을 완료했고 커밋까지 반영했습니다."
> **커밋:** `abc1234` `feat: 로그인 폼 유효성 검사 로직 추가`
---

## 5. 기술 스택 원칙

* **React:** 함수형 컴포넌트와 Hooks를 우선 사용합니다.
* **TypeScript:** 모든 데이터와 함수의 타입을 엄격히 정의하며 `any` 사용을 지양합니다.
* **Code Quality:** 가독성과 재사용성을 고려한 클린 코드를 작성합니다.

---

이 문서는 에이전트가 작업을 시작할 때 가장 먼저 참조해야 하는 **최우선 가이드**입니다. 모든 작업은 이 흐름에 따라 체계적으로 진행되어야 합니다.
